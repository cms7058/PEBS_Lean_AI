# 8D 问题解决方法

8D（Eight Disciplines）由福特公司在 1980 年代制定，是处理**客户投诉、质量异常、批次问题**的标准化结构化方法。在汽车、电子、制造业普遍应用，是供应商质量管理的通用语言。

## 8 个步骤（D1–D8）

### D1：组建团队（Team）
- 5–8 人跨职能（生产、质量、工程、设计、供应商、客户接口）
- 指定团队组长（Champion）和记录人
- 团队成员角色与职责清晰

### D2：问题描述（Problem Description）
- 用 5W2H 描述：Who / What / When / Where / Why / How / How many
- 量化：缺陷率、批次量、影响产品序列号
- 一句话能复述：在什么场景、什么部件、什么现象、什么后果
- 避免："质量不好""偶尔出问题"等模糊表述

### D3：临时遏制措施（Interim Containment Action, ICA）
- 立即保护客户：召回库存、加严检验、隔离批次
- 24–48 小时内执行
- ICA 必须可被验证（如：每件加 100% 全检）

### D4：根本原因分析（Root Cause Analysis）
- **流出原因**：为什么没被检出？（检验体系）
- **发生原因**：为什么会产生？（设计/工艺/管理）
- 工具：5 Why、鱼骨图、是 / 不是分析（Is / Is Not）、过程图
- 必须用数据验证根因（如重现试验）

### D5：永久纠正措施（Permanent Corrective Action, PCA）
- 针对根因设计长期对策
- 区分流出对策与发生对策
- 评估对策的有效性（小批量试运行）
- 评估对策的副作用

### D6：实施 PCA 与验证（Implement PCA）
- 把 PCA 落地到生产现场
- 验证连续 N 批 / N 天无再发
- 撤除 D3 临时措施
- 验证数据归档

### D7：预防再发（Prevent Recurrence）
- 修订标准书、FMEA、控制计划、检验指导书
- **横向展开**：其他相似产品 / 工序 / 工厂是否有同类风险
- 培训与考核

### D8：团队表彰（Team Congratulations）
- 总结经验教训
- 公开表彰团队
- 文档归档（知识库 / Lessons Learned）

## D 与 D 之间的迭代

- D3 临时措施可在 D2 完成后立即执行，不必等 D4
- D4 与 D5 常并行：边追根因边设计对策
- D6 验证失败要回到 D4 或 D5

## 8D 报告模板要素

| 字段 | 内容 |
|---|---|
| 客户 / 内部编号 | 客户提供的投诉编号 + 内部跟踪号 |
| 责任部门 | 主责部门 + 配合部门 |
| 严重等级 | A（停线）/ B（影响出货）/ C（一般） |
| 团队成员 | 姓名 + 角色 |
| D1–D8 内容 | 每段含决策依据、责任人、完成日期 |
| 附件 | 分析数据、试验记录、客户确认函 |

## 8D 与其他方法的对比

| 方法 | 触发场景 | 周期 | 框架 |
|---|---|---|---|
| **8D** | 客户投诉、批次质量问题 | 数天–数周 | D1–D8 |
| **A3** | 内部改善、跨部门项目 | 数周 | 1 张 A3 |
| **DMAIC** | 慢性变异、6σ 项目 | 3–6 月 | D-M-A-I-C |
| **Kaizen Event** | 现场快速改善 | 3–5 天 | PDCA |

## 常见误区

- D3 与 D5 混淆——把临时措施当永久措施
- D4 只问发生原因，忽略流出原因——客户继续受影响
- D7 只改本品种，不横展——同类问题重复发生
- 8D 报告写完即归档，无标准化更新——纸面工程

## 实施建议

- 客户/内部 8D 模板统一
- 重大问题 24h 内启动 D1–D3
- 用 PDM / QMS 系统跟踪每个 D 的完成状态
- 定期复盘 8D 库，提炼系统性问题
